# 2026-04-05 Agent Notes

## 错误复盘 1

- 任务背景：修复“`a -> b -> c` 中继链在 `b` 满员时会吞兵”的持续出兵逻辑。
- 错误现象：把规则错误实现成“只有 `overflow_count` 才继续触发 `b -> c` 调度”，遗漏了“运入本身也是新增士兵事件”。
- 根因分析：把“结果状态（溢出多少）”误当成“事件语义（新增几名士兵）”；先整包并城再补派，丢失了逐兵到达的时序。
- 正确做法：友军运兵到达同阵营城市时，要逐兵结算；每 1 名到达士兵都立即触发 1 次持续调度。这里的“新增士兵”既包括本地产兵，也包括外部运入。
- 防再犯措施：
  - 以后遇到“事件驱动”规则时，先写出事件源和触发次数，再决定状态字段，不再直接从最终聚合结果反推事件次数。
  - 先补一条能表达时序的回归测试，再改实现，避免再次把逐步过程做成事后补偿。

## 错误复盘 2

- 任务背景：按用户要求优化游戏 UI，用户明确提出“城市改成 emoji / AI 图 / 网上素材三选一里的一种也可以”，并已在设计阶段选择了偏 emoji 的方向。
- 错误现象：实现阶段为了规避当前字体链路下 emoji 缺字风险，把默认城市展示收成了“短文本 + 程序化底座”，导致用户实际看到的仍然像小圆形据点，而不是预期中的 emoji 城市。
- 根因分析：我把“工程兜底”和“用户第一视觉目标”主次颠倒了。虽然识别到默认 emoji 在现有字体链路下有风险，但没有在实现前把这个取舍重新抛回给用户确认，而是擅自把 fallback 提升成默认方案，结果偏离了用户最关心的视觉结果。
- 正确做法：当默认实现会明显偏离用户已经确认的视觉方案时，必须先明确告知冲突点，并让用户在“保留 emoji 目标、接受潜在缺字风险”与“改为保守默认方案”之间做选择；不能自己替用户做审美优先级决策。
- 防再犯措施：
  - 以后凡是“设计已确认，但实现发现资源/平台限制”这类情况，先停下来同步冲突，而不是直接把降级方案做成默认。
  - 对视觉类需求，最终交付标准优先看“用户第一眼看到的效果是否匹配承诺”，再处理工程保守性；若二者冲突，必须显式告知并获得确认。
