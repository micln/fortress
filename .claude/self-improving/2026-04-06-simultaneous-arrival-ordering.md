# 任务背景
- 场景：多势力部队同帧进入同一城市（尤其是中立城）时，玩家观察到结果不稳定，且升级后守军会出现“瞬间跳高”的违和感。

# 错误现象
- 把问题直觉归因为“并发/多线程”，但实际项目主循环是单线程。
- 旧实现按 `attack`/`launch_order` 串行逐条到达结算，导致同帧同城结果对顺序敏感。

# 根因分析
- 同帧到达没有按“目标城批量聚合”处理。
- 第一支到达部队会先改写城市归属，后续同帧到达会基于被改写后的状态继续结算，放大顺序偏差。

# 正确做法
- 同帧按目标城市分组：先聚合 `attack` 到达，再一次性结算。
- 结算步骤：守方同阵营到达先并入守军；多攻方先同帧互耗；仅让幸存攻方与守军做一次攻城结算。

# 防再犯措施
- 针对“同帧多方入城”和“守方同帧增援+进攻”新增回归测试。
- 以后遇到“看起来像并发”的战斗异常，先检查主循环结算顺序与批处理边界，而不是先假设多线程问题。

# 可复用经验
- 规则层若存在“同帧多事件写同一状态”，默认优先采用“先聚合后结算”而非“到达即写入”。

# 错误复盘（追加）
- 任务背景：修复“满员无持续任务时产能积压，升级后下一秒人数异常暴涨”问题，并补回归测试。
- 错误现象：新增测试初版参数不精确，注释目标与断言结果一度不一致。
- 根因分析：先写了结论，再快速填了参数，漏做 `rate * delta` 的中间量核对。
- 正确做法：改为 `production_rate=1.0` 且累计 `3.2s`，再验证丢弃整数积压后仅保留小数进度。
- 防再犯措施：今后所有数值型测试在提交前都手算关键中间量，确保“注释场景、输入参数、断言值”三者一致。
