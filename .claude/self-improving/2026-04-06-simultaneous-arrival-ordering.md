# 任务背景
- 场景：多势力部队同帧进入同一城市（尤其是中立城）时，玩家观察到结果不稳定，且升级后守军会出现“瞬间跳高”的违和感。

# 错误现象
- 把问题直觉归因为“并发/多线程”，但实际项目主循环是单线程。
- 旧实现按 `attack`/`launch_order` 串行逐条到达结算，导致同帧同城结果对顺序敏感。

# 根因分析
- 同帧到达没有按“目标城批量聚合”处理。
- 第一支到达部队会先改写城市归属，后续同帧到达会基于被改写后的状态继续结算，放大顺序偏差。

# 正确做法
- 同帧按目标城市分组：先聚合 `attack` 到达，再一次性结算。
- 结算步骤：守方同阵营到达先并入守军；多攻方先同帧互耗；仅让幸存攻方与守军做一次攻城结算。

# 防再犯措施
- 针对“同帧多方入城”和“守方同帧增援+进攻”新增回归测试。
- 以后遇到“看起来像并发”的战斗异常，先检查主循环结算顺序与批处理边界，而不是先假设多线程问题。

# 可复用经验
- 规则层若存在“同帧多事件写同一状态”，默认优先采用“先聚合后结算”而非“到达即写入”。

# 错误复盘（追加）
- 任务背景：执行一次小型 UI 调整（城市图标与字体大小）。
- 错误现象：误把 `apply_patch` 通过 `exec_command` 调用，触发工具层告警。

## 追加：数值膨胀改造经验（2026-04-06）

- 任务背景：执行“全量数值 x10 膨胀”，涉及 domain/application/presentation 与测试基线同步。
- 错误现象：批量替换脚本第一次因 shell 引号与换行参数处理错误，未正确改动目标文件。
- 根因分析：对 `perl -pi -e` 的引号转义和多文件参数展开预估不足，直接把换行列表当成单参数传入。
- 正确做法：优先使用 `xargs` 逐文件执行批量替换；替换后立即抽样 `sed` 校验关键文件。
- 防再犯措施：后续批量改值统一采用“执行前列文件 -> 执行后抽样 -> 运行测试”的三步法，不再一次性盲改。

# 错误复盘（追加）
- 任务背景：为“先到城先结算”补充近城优先规则（进入目标城到达圈后不再参与路上互撞）。
- 错误现象：第一次补丁把 `_resolve_marching_collisions()` 内部缩进打乱，导致分支层级异常风险。

- 防再犯措施：以后在 GDScript 的嵌套循环里改逻辑时，固定执行“补丁 -> 行号检查 -> 测试”三连，减少缩进类回归。
