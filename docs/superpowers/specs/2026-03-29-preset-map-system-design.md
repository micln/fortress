# Preset Map System Design

**Date:** 2026-03-29

## Goal

为当前 `Fortress War` 原型新增“预设地图装载”能力，在不破坏现有 `prototype_*` 主循环的前提下，让游戏可以从一张手工定义的中国风抽象战略地图开局，用更明确的空间结构替代当前纯随机地图，提升中后期的战略性与战术杠杆。

## Problem

当前原型的核心闭环已经成立，但玩法体验存在两个直接问题：

1. 对局中后期容易退化为“谁城多、谁兵多、谁就赢”。
2. 玩家能施加影响的变量偏少，开局位置和早期滚雪球经常主导整局结果。

这说明当前战场空间还不够有意义。地图虽然支持大地图拖拽和道路连接，但更多承担的是“单位分布画布”的职责，而不是“战略结构载体”的职责。只在现有随机地图上继续叠加规则，很可能会把复杂度做高，却仍然无法解决策略空间不足的问题。

## Decision

第一阶段先不追求真实历史地图还原，而是采用“玩法优先、历史风格辅助”的预设地图方案：

- 地图以中国古代区域和城池命名为风格来源。
- 城市位置、连线和区域结构优先服务玩法，而不是服务史实比例。
- 第一张图规模控制在 12 到 18 座城市之间。
- 第一阶段不做地图选择 UI，直接在代码中写死加载第一张预设地图。
- 当前随机地图系统保留，不删除，作为后续可并存的另一种地图来源。

这个选择的核心理由是：先把“战场结构”做出来，再谈更高精度的历史地图还原。后续版本可以逐步从 C（玩法优先）向 B（历史氛围 + 抽象）再向 A（更高真实度）靠拢。

## Non-Goals

第一阶段明确不做以下内容：

- 不新增地图选择界面
- 不实现欧洲地图
- 不实现真实历史边界、真实道路或严肃考据地图
- 不新增技能、科技、资源、兵种等复杂系统
- 不重写现有战斗、运兵、AI、产兵规则
- 不删除当前随机地图生成器

## Design Principles

### 1. 先做棋盘，再做规则

当前优先级不是给玩家更多按钮，而是给玩家更有意义的空间。第一张预设图必须能让玩家在自然试玩中感受到：

- 中路争夺有价值
- 侧翼绕后有价值
- 瓶颈防守有价值
- 后方发育区有价值

换句话说，地图不是背景，而是决策结构。

### 2. 架构上支持多地图来源

虽然第一阶段只落地一张写死的预设图，但系统边界不能做成“一次性硬编码”。主场景需要能够从统一接口拿到一局地图数据，而不关心它来自：

- 随机生成器
- 预设地图模板
- 未来的 UI 选择结果

这可以保证后续接入第二张图、欧洲图、半随机模板时不需要重做主流程。

### 3. 不把地图模板逻辑塞进表现层

`prototype_main_game.gd` 负责流程编排和场景装配，不应该持有整张地图的原始静态定义。地图模板应留在 `application` 层，由主场景在开局时调用。

### 4. 保持现有规则闭环不变

本次设计的目标是“替换地图输入”，不是“改写规则系统”。现有城市状态、进攻、运兵、产兵、AI 决策、HUD、行军单位逻辑应尽量复用。只有在预设地图接入后，未来才继续设计高价值战略支点和区域特殊机制。

## Architecture

### Current State

当前主流程中，`prototype_main_game.gd` 会在 `_start_new_match()` 时调用 `prototype_map_generator.gd` 生成城市数组，然后继续复用后续已有流程：

1. 生成城市状态数组
2. 生成城市视图
3. 玩家和 AI 下令
4. 行军推进
5. 攻城、遭遇战、运兵、产兵、胜负判断

### Target State

主流程改为“通过统一地图来源接口拿到城市数组”，具体来源先写死为第一张预设地图。

建议保持以下职责边界：

- `scripts/domain/`
  - 继续负责城市状态对象与领域规则
- `scripts/application/`
  - 保留随机地图生成器
  - 新增预设地图模板定义与装载服务
  - 负责把模板数据转换成现有 `PrototypeCityState` 数组
- `scripts/presentation/`
  - 继续负责主场景、HUD、输入和可视化
  - 只决定“本局使用哪种地图来源”，不持有大段模板数据

### Phase 1 Scope Reduction

为了避免第一阶段过度设计，Phase 1 只引入“两件套”：

- `prototype_preset_map_definition.gd`
  - 描述第一张预设地图的静态数据结构
- `prototype_preset_map_loader.gd`
  - 把模板数据转换成当前运行时城市状态数组

随机地图生成器 `prototype_map_generator.gd` 暂时保留，不与模板系统混写成单个大文件。

`catalog` 或“多地图注册表”留到 Phase 2，再和第二张地图、地图切换能力一起引入。

### Runtime Contract

统一地图来源接口在第一阶段至少要收敛到同一个最小 contract，避免实现阶段反复返工。

推荐 contract 如下：

`build_map(match_config: Dictionary, map_world_size: Vector2, random: RandomNumberGenerator) -> Array`

其中：

- `match_config`
  - 至少包含当前总方数、AI 难度、AI 风格
  - 第一阶段里预设图主要消费总方数字段
- `map_world_size`
  - 当前对局的地图世界尺寸
- `random`
  - 预留给随机地图或未来带随机出生变体的预设图使用

返回值：

- `Array`，元素为当前主流程可直接消费的 `PrototypeCityState`

错误暴露方式：

- 对配置错误、模板错误、连通性错误直接返回失败并输出明确错误信息
- 不允许“静默降级成空地图”或“偷偷回退到随机图”

## Data Model

第一阶段的模板数据至少需要描述以下信息：

- 地图元信息
  - 地图 ID
  - 地图名称
  - 地图主题说明
  - 设计画布尺寸
  - 支持总方数列表
- 城市定义
  - 城市 ID
  - 城市名
  - 设计画布坐标
  - 初始兵力
  - 初始等级
  - 初始防御
  - 初始产能
- 道路定义
  - 哪些城市彼此相连
- 出生配置
  - 按总方数划分的出生城集合
  - 每种总方数下，明确玩家出生点和 AI 出生点列表

第一阶段可选但不强制的数据：

- 区域标签
- 战略标签
- 节点类型（关口、腹地、枢纽）

这些字段可以预留结构，但不要求第一版就进入实际玩法逻辑。

## Faction Count Rule

当前游戏已经支持开局选择 `2` 到 `5` 方，因此 Phase 1 不能通过“写死预设图”破坏这一现有能力。

第一阶段的明确规则如下：

- 第一张预设图继续支持 `2` 到 `5` 方开局
- 模板中必须提供 `spawn_sets_by_faction_count`
- loader 在构建地图时，按当前 `_player_count` 选择对应出生配置
- 未被当前方数使用的出生城，在该局中回落为中立城市
- 开始面板中的总方数选择暂时保留，不隐藏、不锁定

这样可以保持现有 UI 行为不变，也避免第一阶段去改说明面板和文档中的方数配置逻辑。

## Coordinate Contract

当前运行时 `_map_world_size` 会按视口动态计算，因此模板不能直接存“依赖当前屏幕的最终世界坐标”。第一阶段采用固定设计画布坐标契约：

- 每张预设地图定义自己的 design canvas，例如 `1200x2200`
- 模板内所有城市坐标都使用该 design canvas 上的绝对坐标
- loader 在开局时根据当前 `map_world_size` 把设计坐标映射到实际世界坐标
- 映射完成后，再执行边界、重叠、连通性等校验

这个契约的好处是：

- 地图编辑时有稳定坐标系
- 不依赖当前设备分辨率直接写模板
- 仍然兼容当前大地图自适应和拖拽逻辑

第一阶段不使用 `0..1` 归一化坐标，也不直接在模板里写“已经映射好的最终世界坐标”。

## First Preset Map

### Theme

第一张图采用“中国风抽象战略图”，重点不是还原真实全国地图，而是用中国古代战略地理的感觉来塑造战场结构。

### Size

- 城市数量：12 到 18 座
- 推荐起始规模：14 到 16 座更稳

### Structural Targets

这张图必须明确具备以下结构特征：

- 1 个中央高冲突区
  - 开局后多方较快接触
  - 是最容易发生持续争夺的区域
- 2 条侧翼通路
  - 允许绕开中央主战场
  - 让“侧翼推进”和“回防判断”变得重要
- 1 到 2 个瓶颈点
  - 适合少兵防守或形成卡点
  - 让局部地理结构能放大战术价值
- 1 到 2 个后方安全区
  - 适合作为发育、补兵、运兵中转点
- 若干主干连接城市
  - 让交通枢纽有天然战略价值

### Balance Targets

虽然地图不需要完全对称，但必须避免：

- 某一出生点天然控制中央枢纽
- 某一出生点明显更容易滚雪球
- 某一侧几乎没有可防守节点

允许“非对称”，但不允许“天然碾压”。

### Spawn Targets

为了兼容当前 `2` 到 `5` 方玩法，第一张地图需要至少定义以下出生约束：

- `2 方`
  - 玩家与单个 AI 分居地图两侧，中央争夺区位于中段
- `3 方`
  - 三方出生点应能在中路形成三角争夺，而不是两家先互撞、一家安稳发育
- `4 方`
  - 四方应覆盖中路与侧翼，不允许某一方独占后方安全区
- `5 方`
  - 保持当前默认配置可玩，确保中央不会被单一起点天然锁死

### Naming Style

命名上优先采用中国古代风格城市/区域意象，但不要求严格对照史实位置。例如可以使用：

- 长安
- 洛阳
- 邺城
- 汴州
- 襄阳
- 寿春
- 成都
- 建业

如果玩法需要，也可以使用带有区域感的抽象节点名，而不是一味追求全部都是真实古城。

## Gameplay Impact Expectations

这次改动完成后，理想的试玩感受应该出现以下变化：

- 玩家开始主动争中央节点，而不是只按离得最近扩张
- 玩家会把部分兵力留在瓶颈和交通节点，而不是一味向外推
- 绕后路线和回防节奏变得更重要
- 领先方虽然仍然更强，但需要判断“控中路”还是“扫侧翼”
- 落后方在局部可以利用路径、节点和集中兵力打出反扑

如果预设地图上线后，玩家仍然只是在更漂亮的棋盘上重复“兵多城多必赢”，那说明后续还需要继续加强战略支点或区域机制。

## Integration Plan

### Phase 1

仅完成以下能力：

- 新增预设地图模板结构
- 新增第一张中国风预设地图
- 主场景开局时写死使用这张预设地图
- 让现有主循环完整跑通
- 保留现有开始面板中的总方数选择，并让预设图按方数切换出生配置

### Phase 2

在第一张图跑通并试玩验证后，再考虑：

- 补第二张预设图
- 加回随机地图与预设图的切换
- 设计高价值战略支点或区域效果

### Phase 3

再往后才考虑：

- 更强历史氛围地图
- 欧洲地图
- 地图选择 UI
- 半随机模板

## Error Handling

预设地图系统需要能显式暴露以下错误，而不是静默容忍：

- 城市 ID 重复
- 连线引用不存在的城市
- 城市邻接关系不对称
- 指定总方数缺少出生配置
- 出生配置中的城市重复或引用不存在
- 设计画布坐标映射后越界或重叠严重
- 地图整体不连通

第一阶段即使不把所有错误都做成完整测试，也应至少在模板装载时提供明确校验，避免出现“地图能开局，但局内行为异常”的隐性问题。

## Testing Strategy

### Automated

建议新增或扩展以下测试覆盖：

- 预设地图模板可以成功转换成 `PrototypeCityState` 数组
- `2` 到 `5` 方都能选到合法出生配置
- 预设地图城市数量符合预期
- 所有连线合法且邻接关系完整
- 地图整体连通
- 运行入口仍指向 `prototype_main.tscn`
- 主流程可以从预设地图正常开局

### Manual

至少进行以下试玩验证：

- 开局是否进入预设图而不是随机图
- 切换 `2/3/4/5` 方后，出生点是否随之变化且没有非法重叠占位
- 中路、侧翼、瓶颈和后方区域是否肉眼可识别
- 多方 AI 是否会围绕中心区与侧翼自然形成战线
- 玩家是否开始更频繁做“卡点、绕后、回防”判断
- 大地图拖拽、HUD、出兵面板、暂停等现有表现层逻辑是否仍然正常

### Phase 1 Acceptance Constraints

为了避免“地图结构加大了复杂度，却没有提升可玩性”，Phase 1 额外增加以下约束：

- 默认 `5` 方局在正常试玩中，应能较早形成中央接触，而不是长时间各自闷头发育
- 城市数量提升到 12 到 18 后，城市标签在常见移动端纵向视口下不应出现大面积持续重叠
- AI 在开局后应能按现有节奏完成首次扩张，不应因地图结构变化导致明显长时间停滞

## Success Criteria

第一阶段可以视为成功的标准是：

- 系统层面已支持预设地图作为独立地图来源
- 第一张中型中国风抽象战略图可以稳定开局并跑完整局
- 预设图在 `2` 到 `5` 方配置下都能选择合法出生点
- 现有规则系统不需要大改即可兼容预设图
- 实际试玩中，地图位置和路线开始明显影响决策质量
- 后续继续做战略支点、区域机制、第二张地图时，不需要推翻这次架构

## Open Questions

以下问题留到实现或后续版本再定：

- 第一张地图是否固定为 14、15 还是 16 座城市
- 初始出生点采用完全固定还是小范围变体
- 是否在第一张图里提前标记“未来战略支点”字段
- 是否在模板层就加入区域标签，还是等第二阶段再补

## Recommendation

按当前项目状态，推荐执行顺序如下：

1. 先完成预设地图模板和装载架构
2. 再接入第一张中国风抽象中型地图
3. 用现有规则直接试玩验证策略性变化
4. 只有确认地图结构已经提升决策空间后，再继续设计战略支点与区域机制

这是当前最稳、最符合项目阶段的推进方式。
