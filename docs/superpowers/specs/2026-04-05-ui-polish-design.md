# 2026-04-05 UI Polish Design

## 背景

当前 `prototype_*` 主链路已经具备完整的可玩原型，但视觉表现仍停留在“规则优先”的简化阶段，主要问题集中在两处：

- 城市图标虽然已经做了程序化中式城池轮廓，但整体仍偏硬、偏素，类型差异主要依赖文字属性行而不是第一眼视觉识别。
- 战场背景目前以单一绿色矩形为底，只叠加少量固定矩形土块和等距斜线，导致整张地图观感过平、过整齐。

用户本轮需求不是重做美术体系，也不是引入大量外部资源，而是做一次轻量但明显可感知的 UI 优化，让城市更好看、地图稍微细腻一些，同时保持当前原型的可读性和低接入成本。

## 目标

本次设计只解决以下目标：

1. 让城市节点在不引入外部图片资源的前提下，获得更强的“古代城池感”和类型辨识度。
2. 让战场背景从“纯绿大块”升级为更有层次的地表，但不削弱道路、城市、行军单位的可读性。
3. 保持当前 `presentation -> application -> domain` 分层，不把视觉增强扩散到规则层。
4. 保持移动端 / Web / 桌面端的交互稳定性，不影响拖拽、缩放、选城命中和现有测试入口。

## 非目标

以下内容明确不在本次范围内：

- 不改战斗、出兵、AI、产兵、升级等规则。
- 不改预设地图布局、道路连接关系、城市坐标和节点类型定义。
- 不引入外部网络素材、在线资源或运行时图片下载逻辑。
- 不新增 TileMap、Shader、Noise 资源或完整美术资源管线。
- 不重做 HUD、说明面板、按钮布局，只处理城市图标和战场背景的观感提升。

## 方案概览

本次采用 `C-lite` 方案：

- 城市主体从“纯程序化城池主体”调整为“emoji 主体 + 程序化底座与状态装饰”。
- 城市类型通过少量 emoji 差异直接传达：`normal=🏯`、`pass=🏰`、`hub=⛩️`、`heartland=🏯`，其中 `heartland` 不单独换成另一类建筑，而是通过固定的“核心外圈 + 底部徽记”表达“核心据点”。
- 战场背景继续使用 `prototype_main_game.gd` 中的即时绘制方案，但升级为四层程序化地表：主草地、土地色块、草纹/地形纹、低对比氛围层。

这样做的原因是：

- 能满足用户对“古代城池感 + 类型更易区分”的需求。
- 不需要新增贴图资源，不会引入 Web 导出资源一致性风险。
- 改动集中在表现层，且基本落在现有 `_draw()` 与模板节点链路上，集成风险低。

## 城市视觉设计

### 1. 视觉结构

每个城市节点保持三层信息结构：

1. 顶层：emoji 主体，承担第一眼识别
2. 中层：程序化底座、阵营描边、选中高亮、轻量承托装饰
3. 底层：现有文字信息（名称、兵力、属性）

emoji 不是唯一信息来源，只负责快速识别；当前 `NameLabel`、`SoldierLabel`、`AttrLabel` 继续承担稳定的信息阅读职责。

### 2. 节点类型到 emoji 的映射

- `normal` -> `🏯`
- `pass` -> `🏰`
- `hub` -> `⛩️`
- `heartland` -> `🏯`

`heartland` 不单纯依赖 emoji 差异，而是必须同时具备以下两个固定特征：

- 一圈比 `normal` 更醒目的核心外圈描边
- emoji 下方一个固定的菱形或印章式徽记

这样可以在不增加第四种主要建筑 emoji 的前提下，仍然稳定地区分 `heartland` 与 `normal`。

### 3. 状态表达

保留并强化以下状态表达：

- 阵营归属：继续使用阵营色作为底座或描边主色
- 选中态：继续保留明显高亮框，但需与 emoji 主体协调
- 等级感：不再通过旗幡长度做主识别，可改为底座徽记、描边厚度或小型点缀

### 4. 布局原则

- 碰撞盒仍保持原有矩形稳定范围，不跟随 emoji 轮廓收缩，避免点击区域变窄。
- emoji 与文字之间保留足够垂直间距，避免窄屏下重叠。
- 表现层必须继续由主场景统一传入屏幕坐标，不在城市视图中重新计算世界坐标。

### 5. emoji 降级策略

emoji 只作为主识别层，不允许成为单点失败依赖。降级策略固定为：

- 如果目标环境下 emoji 显示为空白、方框、严重错位，必须回退为短文本标识或纯程序化符号。
- 回退后的最小语义集合为：
  - `normal` -> `城`
  - `pass` -> `关`
  - `hub` -> `枢`
  - `heartland` -> `腹`
- 即使 emoji 正常显示，也必须保留底座、描边、徽记与属性文本，确保识别不依赖单一字体能力。

## 地图背景设计

### 1. 总体原则

背景只做“更细腻”，不做“更复杂”。核心要求是：

- 城市、道路、行军单位永远比地表更醒目。
- 地表细节只能提供氛围，不能形成视觉干扰或伪交互元素。
- 所有元素继续跟随现有 `_map_offset` 与 `_map_zoom` 换算，无需新增独立背景节点。

### 2. 四层地表结构

#### 第一层：主草地

替换当前单一纯绿色底矩形，改成更有层次的绿色主底，可以通过多块低对比区域叠加实现，而不是整个战场统一一色。

#### 第二层：土地色块

将现有固定矩形土块改成更自然的不规则块面。目标是形成“裸土 / 干地 / 被踩开的土地”观感，而不是几块生硬色板。

#### 第三层：草纹 / 地形纹

保留轻量线性纹理，但不再使用机械等距平行带。新的纹理应更自然，可采用分段斜纹、短带状纹理或局部方向变化。

#### 第四层：低对比氛围层

增加极轻的林地感、丘陵感或色块阴影，只用于打破平整感，不允许让玩家误以为这里存在障碍、特殊地块或可交互区域。

### 3. 背景约束

为避免实现风格漂移，本次背景增强需要满足以下硬约束：

- 所有背景层都必须绘制在道路、城市和行军单位之下。
- 背景装饰层的透明度应保持低强度，单层 alpha 不高于 `0.22`，避免压过主信息层。
- 不新增会被误认成道路节点、障碍物或可点击对象的封闭高对比图形。
- 不覆盖或削弱道路描边的可见性。

## 实现设计

### 1. 城市节点模板

在 `scenes/main/prototype_main.tscn` 的城市模板节点（旧实现）中增加一个专用于 emoji 的 `Label`。该节点应：

- 使用现有字体资源，避免 Web 缺字链路漂移
- 在城市主体区域居中显示
- 与现有名称、兵力、属性三层文本分开摆放

### 2. 城市视图脚本

在 `scripts/presentation/prototype_city_view.gd` 中：

- 增加 `emoji_label` 引用
- 在 `sync_from_state()` 中根据 `node_type` 更新 emoji
- 将 `_draw()` 从“完整城池主体绘制”调整为“底座 + 阵营色描边 + 选中态 + 轻量承托装饰”
- 继续保留函数头注释，并明确说明调用场景和主要逻辑

### 3. 战场背景绘制

在 `scripts/presentation/prototype_main_game.gd` 中：

- 只改 `_draw_battlefield_background()`
- 如函数体膨胀，可拆出少量表现层 helper，例如：
  - 地块多边形构造
  - 局部纹理绘制
  - 背景装饰层绘制

这些 helper 仍然必须留在 `presentation` 层，不得下沉到 `application` 或 `domain`。

### 4. 注释要求

本次新增或修改的所有函数，包括 `prototype_city_view.gd` 和 `prototype_main_game.gd` 中新增的背景 helper，都必须补齐函数头注释，说明功能、调用场景和主要逻辑。

## 错误处理与兼容性

### 1. emoji 差异风险

不同平台字体对 emoji 造型会有差异，尤其是 Web 与桌面环境。为降低风险：

- 控制 emoji 种类数量，不使用太多冷门符号
- 不将语义完全依赖于 emoji，自始至终保留属性文字和节点标签
- 不按 emoji 实际轮廓计算点击范围
- 一旦发现当前平台 emoji 显示异常，必须启用短文本 / 程序化符号回退，而不是接受空白或方框显示

### 2. 可读性风险

地表细节如果过强，会和道路、兵线打架。必须确保：

- 背景透明度整体偏低
- 道路和行军单位仍为主焦点
- 选中态和阵营色不会被背景吞掉

### 3. 适配风险

所有新增视觉都必须继续服从：

- 当前动态分辨率策略
- 拖拽 / 缩放世界坐标换算
- 移动端和 Web 点击命中稳定性

```

```

### 手工验收

至少验证以下点：

1. 主场景可正常启动
2. 四类节点满足最小识别标准：
   - `normal` 与 `heartland` 虽同用 `🏯`，但 `heartland` 必须同时带有“核心外圈 + 底部徽记”
   - `pass` 与 `hub` 能通过主体符号直接区分
3. 阵营色、选中态、兵力数字仍然清楚
4. 地图明显比当前更细腻，但不抢道路、城市、行军单位视觉焦点
5. 拖拽、缩放、点城、空白取消、升级浮层位置不受影响
6. 窄屏 / 竖屏下 emoji 与文字不重叠、不越界
7. Web 下中文与 emoji 正常显示；若 emoji 显示异常，回退标识能正常显示且不出现方框或空白
8. 缩放后点击命中与视觉位置保持一致

## 取舍总结

本方案的核心取舍是：

- 优先用最小表现层改动换来明显观感提升
- 接受 emoji 的有限平台差异，换取极低接入成本
- 不引入外部贴图资源，保持 Web 与本地一致性
- 不追求“真实地图材质”，只追求“比当前更细腻，但仍清楚可玩”
