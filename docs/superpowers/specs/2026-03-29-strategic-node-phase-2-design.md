# Strategic Node Phase 2 Design

**Date:** 2026-03-29

## Goal

在当前 `Fortress War` 的首张预设地图基础上，引入“静态战略节点”能力，让部分城市具备明确的战略身份，从而放大地图结构对进攻、防守、发育与回防判断的影响。

## Context

`preset map` Phase 1 已经完成：

- 预设地图可作为独立地图来源正常开局
- `2` 到 `5` 方出生配置可用
- 首张中国风抽象战略图已通过自动化与手工验收

当前下一阶段不再优先做“多地图切换”，而是优先验证：给关键城市加入轻量战略身份后，玩家与 AI 是否会更明显地围绕关键点做决策。

## Problem

虽然当前地图结构已经优于纯随机图，但如果所有城市仍然只是“名称、坐标、基础数值”不同的普通节点，那么地图的结构价值仍然主要来自道路关系，而不是来自节点本身的战略定位。

这会带来两个问题：

1. 玩家知道中路和侧翼有区别，但缺乏更明确的“为什么要守这里、抢那里”的反馈。
2. 地图虽然有瓶颈、后方和枢纽感，但这些感受还没有被规则系统轻量放大。

## Decision

Phase 2 先做“静态节点效果”，不做区域占领系统，不做道路地形特效。

首版只定义 3 类节点：

- `PASS`
  - 关口
  - 强调瓶颈防守价值
- `HUB`
  - 枢纽
  - 强调交通/兵源核心价值
- `HEARTLAND`
  - 腹地
  - 强调后方发育与中转价值

节点效果采用“轻量数值修正”策略，而不是引入新的独立子系统。

## Non-Goals

本阶段明确不做：

- 不做区域占领加成
- 不做道路边的特殊效果
- 不做地图选择 UI
- 不做第二张地图
- 不做随机图与预设图切换
- 不重写 AI 主决策框架
- 不新增复杂资源、补给、科技或兵种系统

## Design Principles

### 1. 节点效果必须服务现有地图结构

节点类型不是为了给城市贴标签，而是为了放大当前已存在的地理结构：

- 关口应对应瓶颈或必经路口
- 枢纽应对应中央争夺区或多路连接点
- 腹地应对应相对安全、适合发育和回防中转的位置

### 2. 第一版只做可解释、可测试的轻量规则

节点效果必须满足：

- 玩家能直观理解
- 文案容易说明
- 自动化测试容易覆盖
- 不需要改写主循环

因此第一版不做“特殊行动规则”，只做基础属性增益。

### 3. 规则落在 domain / application，不落在 presentation

- `domain`
  - 持有节点类型与运行时城市属性
- `application`
  - 从预设地图定义装配节点类型，必要时应用对应数值修正
- `presentation`
  - 只负责显示节点类型和提示文案

### 4. 策略提升优先于花哨效果

这次目标不是做一套酷炫但难维护的新机制，而是验证以下体验是否增强：

- 玩家更愿意守关口
- 玩家更愿意抢枢纽
- 腹地更自然地成为补兵和中转点

## Node Types

### PASS

用途：

- 放在易守难攻的瓶颈节点
- 让少兵防守更有价值

首版效果：

- 城市基础防御 `+1`

### HUB

用途：

- 放在中路或多方向交通节点
- 让争夺核心节点更值得

首版效果：

- 城市基础产能 `+0.2`

### HEARTLAND

用途：

- 放在相对后方、安全的发育点
- 让玩家和 AI 更自然地形成“前线 / 后方”分工

首版效果：

- 城市初始兵力 `+2`

## Data Model

Phase 2 在预设地图定义里为城市新增 `node_type` 字段。

推荐值：

- `"normal"`
- `"pass"`
- `"hub"`
- `"heartland"`

运行时城市状态也需要持有该字段，以便：

- UI 显示城市战略身份
- 测试验证节点类型是否正确装配
- 后续继续扩展节点效果时不需要再改装配契约

## Runtime Contract

Phase 2 继续复用现有地图装配入口：

`build_map(match_config: Dictionary, map_world_size: Vector2, random: RandomNumberGenerator) -> Array`

变化点：

- `PrototypePresetMapDefinition` 输出的城市定义会包含 `node_type`
- `PrototypePresetMapLoader` 在构建 `PrototypeCityState` 前应用节点效果修正
- `PrototypeCityState` 将显式持有 `node_type`

## Integration Plan

### Domain

在 `PrototypeCityState` 中增加：

- 节点类型字段
- 节点类型常量
- 节点类型展示名辅助函数（若需要）

但不把具体“数值修正逻辑”塞到表现层。

### Application

在 `PrototypePresetMapDefinition` 中：

- 为首张图上少量关键节点标注 `node_type`

在 `PrototypePresetMapLoader` 中：

- 校验 `node_type` 是否合法
- 根据节点类型修正初始数值
- 把最终节点类型写入运行时城市状态

### Presentation

在主场景与城市展示相关逻辑中：

- 给城市信息增加节点类型提示
- 不强制做复杂图标系统
- 第一版允许用简短中文标签体现，例如“关口 / 枢纽 / 腹地”

## Suggested First Map Tagging

建议第一张图中的首批战略节点如下：

- `洛阳`
  - `hub`
  - 理由：中路核心争夺点
- `许昌`
  - `hub`
  - 理由：承接中路与下方通路的转运节点
- `长安`
  - `pass`
  - 理由：左上进入中路的关键关口
- `汉中`
  - `pass`
  - 理由：左翼向下与中路联通的瓶颈
- `成都`
  - `heartland`
  - 理由：左下后方发育点
- `交州`
  - `heartland`
  - 理由：最下方相对安全的后方节点

第一版不需要给每座城市都贴标签，保留大量 `normal` 节点更有利于对比体验。

## Error Handling

Phase 2 需要新增以下显式错误校验：

- `node_type` 不是支持值
- 战略节点装配后导致基础属性越界
- 城市定义缺少 `node_type` 时应回落到 `normal`，而不是崩溃

## Testing Strategy

### Automated

新增或扩展以下测试：

- 预设地图关键城市能装配出正确的 `node_type`
- `pass` 会提高防御
- `hub` 会提高产能
- `heartland` 会提高初始兵力
- 未声明 `node_type` 的城市回落为 `normal`
- 现有地图连通、出生配置和入口链路测试继续保持通过

### Manual

重点试玩验证：

- 玩家是否更愿意守 `pass`
- 玩家是否更愿意抢 `hub`
- `heartland` 是否自然成为后方补兵点
- 城市 UI 是否能识别节点类型
- 这些改动是否没有破坏拖拽、HUD、出兵面板、暂停等现有流程

## Acceptance Criteria

Phase 2 第一阶段可以视为成功的标准：

- 首张预设图中已有少量关键城市具备明确战略身份
- 节点效果能通过自动化测试验证
- 玩家在试玩中能感知“关口 / 枢纽 / 腹地”的差异
- 现有主循环、AI 与地图装配链路无需大改即可兼容

## Recommendation

按当前项目阶段，推荐执行顺序如下：

1. 先为 `PrototypeCityState` 与预设地图定义补 `node_type`
2. 再由 loader 应用轻量节点效果
3. 然后补自动化测试锁定行为
4. 最后在 UI 中补最小可感知提示并试玩验证

这是当前最稳、最符合 YAGNI 的推进方式。
