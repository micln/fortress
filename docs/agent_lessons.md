# Agent Lessons

## 2026-03-22

- 不要在同一轮实现里同时保留两套核心模型并混接主场景。此次先后出现了 `City/GameState/Faction` 与另一套原型命名结构，导致场景脚本引用和类型解析互相打架。
- 手写 `.tscn` 前必须先读取仓库中的真实场景结构。此次一开始假定主场景只有 `MapView/Hud`，实际仓库已有 `Cities/UILayer/城市模板节点` 结构，造成节点路径错误。
- Godot 原型阶段优先使用 `preload` 和更稳妥的弱类型跨脚本引用，先保证装配稳定，再逐步加强静态类型。
- 设计 AI 难度和风格时，要把“是否出手”和“偏好打谁”拆成两个维度。此次一开始把保守阈值直接压在最终派兵数上，导致简单防御型在部分局面下完全不行动；修正方式是用源城总体优势决定能否出手，再用评分函数决定更偏向玩家城还是中立城。
- 引入第三方字体或素材前，先验证下载地址和许可证文件路径，再写自动化命令。此次先假设 `notofonts/noto-cjk` 根目录直接提供许可证文本，结果命中 404，虽然字体已下成功，但补许可证多走了一步。
- 新增城市属性或经营操作时，要同步校准旧测试基线。此次引入“防御计入攻城门槛”和“升级消耗驻军”后，若直接沿用旧断言，会把规则 bug 和测试预期漂移混在一起。

## 2026-03-23

- 在做“窗口随尺寸变化自适应”时，不要默认使用 `window/stretch/scale_mode="integer"`。整数缩放只会在缩放倍率跨档位时变化，常见桌面窗口拖拽过程中会表现为“看起来没变化”。需要连续自适应时应优先使用 `fractional`，再结合清晰度回归测试确认效果。
- 需求中出现“不要固定分辨率、按设备尺寸动态决定”时，不能只改项目拉伸参数；还要在运行时把窗口实时尺寸同步到 `Window.content_scale_size`，并监听 `size_changed` 做持续更新。
- 修复移动端 Web 字体过小时，不能把 `window.size`（物理像素）直接写入 `Window.content_scale_size`。在高 DPI 设备上会导致整体 UI 再次缩小；应先按 `DisplayServer.screen_get_scale()` 转为逻辑尺寸再写入。
- 做移动端 Web 字体修复时，不能把高 DPI 换算策略无差别应用到桌面 Web；应增加“小屏条件”分支，否则桌面浏览器可能出现画面黑边或看起来固定比例。
- 当 Godot Web 明确要求 Secure Context 时，不要继续通过修改导出产物去绕过 HTTPS 校验。正确做法是提供稳定的本地 HTTPS 联调脚本，并把“纯 HTTP 不受支持”写进运行文档。
- 用窗口尺寸阈值去猜“是不是手机”不可靠，尤其是高 DPI Web 设备。应优先用“是否触屏设备”来决定是否应用移动端逻辑尺寸换算。
- 移动端底部操作栏不能沿用桌面按钮文案和最小宽度；否则即使弹窗已自适应，底栏也会继续遮挡地图或把关键按钮挤出可视区。应为窄屏单独收缩边距、字号和按钮文案。
- 移动端 HUD 的问题往往不只是“宽度过大”，还包括固定外边距和内边距叠加造成的高度浪费。若目标是最大化战场可视面积，应优先把 HUD 贴边，并删减次要文案，而不是只缩按钮宽度。
- 把“点空白取消选城”放在 `_input()` 后，不能只用单次布尔标记去赌事件顺序。移动端 Web 可能连续派发触摸抬起和兼容鼠标事件，更稳的做法是保留一个很短的取消保护窗口，吞掉后续 1~2 次 release。
- 移动端 Web 的点按问题不一定是坐标换算错，也可能是浏览器在触摸后补发兼容鼠标点击。若城市节点同时监听 `ScreenTouch` 和 `MouseButton`，需要在表现层做去重，否则一次点城会触发两次 `city_pressed`。
- 当日志已经明确显示手机一次轻触会同时进入 `mouse` 和 `touch` 分支时，不要继续围绕坐标系打转。最直接、最稳的修法是在触屏设备上直接禁用城市节点的兼容鼠标点击路径。
- 输入保护窗口如果只靠计数消费，而不区分“同一次手势”和“下一次新手势”，很容易引入“空白取消要点两次”的副作用。保护逻辑必须在新的一次按下开始时立即清空。
- 只修开局弹窗不够，出兵弹窗同样会被固定最小宽度和桌面字号撑爆。移动端弹窗检查要覆盖所有二级对话框，而不是只看首屏 Overlay。
- 当弹窗底部按钮“看不到”时，优先检查是不是**垂直高度溢出**，而不是继续缩底部按钮字体。正确修法通常是让正文区滚动、底部操作按钮固定。
- 当输入问题反复出现且无法靠推断收敛时，应优先加结构化日志，而不是继续猜。至少要同时打印：城市世界/屏幕坐标、主场景收到的全局点击坐标、城市节点收到的局部 touch/mouse 事件，才能区分是坐标错位还是重复事件。

## 2026-03-31

- 给表现层补文案时，不要把不同领域的常量直接混用。此次在 `prototype_city_view.gd` 里一度把 `city.node_type` 和阵营常量做比较，虽然很快修正，但说明我在写最小 UI 逻辑时没有先确认字段语义。后续凡是跨字段比较，先核对“类型维度是否一致”再下笔。

## 2026-04-04

- 给“持续出兵”这类自动化行为复用现有一次性出兵函数时，不能忽略函数的交互副作用。此次一开始直接复用 `_execute_transfer/_execute_attack`，但这两个函数默认会清空 `_selected_city_id`，导致自动出兵触发时把玩家当前选中打断。后续凡是复用执行链路，必须先列出“会改哪些 UI 状态”（选中态、提示文案、音效）再接入自动调度。
- 自动化功能上线时，不能只提供“开关入口”，还要提供“运行中可观察反馈”。此次初版开关文案不够突出，且缺少每秒实际下发统计，玩家难以判断功能是否工作。后续类似功能至少同时提供：明显入口 + 常驻状态反馈（频率/次数/规模）。

## 2026-04-05

- 当“持续出兵”已经成为主玩法时，不能继续把“一次性出兵弹窗”和“持续任务调度”混在同一主场景链路里修补。正确做法是把任务模型和轮转调度独立到 `application`，让主场景只负责选城、展示和复用行军执行链。
- 持续任务的规则边界要在模型层一次定死，而不是分散在多个 UI 回调里。此次重写后把“目标换手不删任务、源城失守删任务、每产 1 兵只调度 1 次、多路线轮转”都收敛到 `PrototypeOrderDispatchService`，后续若再改规则也必须优先改服务和测试，而不是先改按钮逻辑。
